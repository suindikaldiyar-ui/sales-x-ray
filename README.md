# Sales X-Ray 🩻

**Мультитенант SaaS для аналитики и роста продаж** отделов продаж
(автосалоны и др.). Это **фундамент**: аутентификация, мультитенантность с
изоляцией данных по компаниям, роли, схема БД и каркас интерфейса.

Интеграции (amoCRM, Wazzup, Sipuni, Telegram), анализ переписок/звонков и
Telegram-отчёты подключаются поверх этого фундамента на следующих этапах —
сейчас для них подготовлены таблицы, заглушки-коннекторы и формы хранения
ключей.

---

## Стек

- **Next.js 14 (App Router) + TypeScript (strict) + Tailwind CSS** — тёмная
  премиальная аналитическая тема, без сторонних UI-китов.
- **Supabase**: PostgreSQL + Supabase Auth (email + пароль, подтверждение
  почты) + **Row Level Security** для изоляции данных между компаниями.
- Доступ к данным — через **Supabase client (`@supabase/ssr`)**, ключи только
  на сервере.

### Почему Supabase client, а не Prisma

Изоляция тенантов здесь построена на **RLS**, а политики RLS опираются на
`auth.uid()` из JWT Supabase Auth. Supabase client (серверный и браузерный)
автоматически передаёт сессию пользователя в каждый запрос, поэтому **RLS
применяется сама собой** — пользователь физически не может прочитать данные
чужой компании, даже при ошибке в коде приложения.

Prisma подключается к Postgres напрямую (как суперпользователь) и **обходит
RLS** — изоляцию пришлось бы повторно реализовывать в коде приложения, что
менее безопасно. Поэтому:

- **Чтение/запись из приложения** → Supabase client с сессией пользователя
  (RLS включена).
- **Схема БД** → ведётся явно как **версионируемые SQL-миграции** в
  [`supabase/migrations`](supabase/migrations) (никакой «магии» — весь DDL
  виден и ревьюится).
- **Сервисные операции** (будущая фоновая синхронизация) →
  `service_role`-клиент ([`lib/supabase/admin.ts`](lib/supabase/admin.ts),
  `server-only`), который сознательно обходит RLS.

### Мультитенант и роли

- **Organization** — компания-клиент (тенант).
- **Profile** — зеркало `auth.users` (создаётся триггером при регистрации).
- **Membership** — связь User↔Organization с ролью: `OWNER` (владелец),
  `ROP` (РОП), `MOP` (менеджер).
- **Subscription** — план/статус/период (заготовка под биллинг, без оплаты).
- При регистрации создаётся `Organization` + `Membership(OWNER)` + триал —
  через SECURITY DEFINER функцию `create_organization`.
- **OWNER/ROP** приглашают пользователей по email (роль РОП/МОП); приглашённый
  принимает приглашение по ссылке `/invite/<token>`.
- Все данные строго изолированы по `organization_id` через RLS: пользователь
  видит только свои компании.

Подготовлены под будущее (минимальные, но в схеме): `integrations`, `leads`,
`conversations`, `messages`, `calls`, `daily_reports` — чтобы вся переписка,
звонки и сделки сохранялись по организациям.

## Запуск

### 1. Создайте проект в Supabase

1. Зайдите на <https://supabase.com> → **New project**. Задайте имя, пароль
   БД (сохраните его) и регион.
2. Дождитесь инициализации (~1–2 мин).

### 2. Где взять ключи

Откройте проект → **Project Settings**:

| Переменная | Где взять |
| --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | Settings → **API** → *Project URL* |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Settings → **API** → *Project API keys* → `anon` `public` |
| `SUPABASE_SERVICE_ROLE_KEY` | Settings → **API** → *Project API keys* → `service_role` `secret` (только сервер!) |
| `DATABASE_URL` | Settings → **Database** → *Connection string* → **URI** (подставьте пароль БД) |

Скопируйте `.env.example` → `.env.local` и заполните значения:

```bash
copy .env.example .env.local   # Windows
# cp .env.example .env.local    # macOS/Linux
```

### 3. Примените миграции

Миграции лежат в [`supabase/migrations`](supabase/migrations) и применяются
**по порядку** (0001 → 0002 → 0003).

**Вариант А — Supabase CLI (рекомендуется):**

```bash
npm i -g supabase
supabase link --project-ref <YOUR-PROJECT-REF>
supabase db push
```

**Вариант Б — psql напрямую (по `DATABASE_URL`):**

```bash
psql "$DATABASE_URL" -f supabase/migrations/0001_schema.sql
psql "$DATABASE_URL" -f supabase/migrations/0002_functions.sql
psql "$DATABASE_URL" -f supabase/migrations/0003_rls.sql
```

**Вариант В — SQL Editor в дашборде Supabase:** вставьте содержимое файлов
по очереди и выполните.

### 4. Настройте Auth

- **Authentication → Providers → Email**: включите (по умолчанию включён).
  Подтверждение почты («Confirm email») оставьте включённым — приложение
  обрабатывает сценарий с письмом-подтверждением.
- **Authentication → URL Configuration → Redirect URLs**: добавьте
  `http://localhost:3000/auth/callback` (и адрес продакшена позже).

### 5. Установите зависимости и запустите

```bash
npm install
npm run dev
```

Откройте <http://localhost:3000> → **Регистрация** → подтвердите почту →
создайте компанию → (опционально) сохраните токен amoCRM → дашборд.

Сборка продакшен-версии:

```bash
npm run build
npm start
```

## Переменные окружения

См. [`.env.example`](.env.example). Кратко:

- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` — публичные,
  используются клиентским SDK (запросы под RLS).
- `SUPABASE_SERVICE_ROLE_KEY` — **только сервер**, обходит RLS, импортируется
  лишь в `lib/supabase/admin.ts` (`server-only`).
- `DATABASE_URL` — **только для применения миграций** (psql/CLI); само
  приложение работает через SDK, а не по этой строке.
- `NEXT_PUBLIC_SITE_URL` — базовый URL для ссылок подтверждения/приглашений.
- `SYNC_WINDOW_DAYS` — окно обычной синхронизации amoCRM в днях (по умолчанию
  30); «Полная история» всегда тянет 365.

## Интеграция amoCRM (реальные данные)

amoCRM подключается **per-organization** и питает дашборд и воронку реальными
цифрами.

### Как подключить

1. В amoCRM: **Настройки → Интеграции → Создать интеграцию** (внешняя) →
   вкладка **«Ключи и доступы»** → выдайте права на чтение сделок/воронок/
   событий и **сгенерируйте долгоживущий токен** (живёт до 1 года).
2. В Sales X-Ray: **Интеграции → amoCRM** → вставьте адрес аккаунта
   (`https://mycompany.amocrm.ru`) и токен → **Сохранить и подключить**.
3. Нажмите **Синхронизировать** (на карточке amoCRM или на дашборде).

### Что происходит при синхронизации

`POST /api/sync/amocrm` (только OWNER/ROP) тянет данные через REST API v4 и
сохраняет их по `organization_id` (через клиент пользователя — **RLS
соблюдается**):

- **Воронки и этапы** → `amocrm_pipelines` / `amocrm_stages`. Порядок этапов
  хранится как **позиционный `rank` (1..N по открытым этапам)** — не по полю
  `sort` amoCRM (там большие произвольные числа).
- **Сделки** → `leads` c `pipeline_external_id`, текущим этапом, ценой,
  `is_won/is_lost`, причиной потери.
- **История переходов** (`GET /api/v4/events`, `lead_status_changed`)
  реконструирует **достигнутый этап** (`reached_rank`) каждой сделки — чтобы
  слитые («потерянные») сделки не схлопывались на первый этап, а учитывались в
  когортной доходимости.

Технически клиент (`lib/integrations/amocrm.ts`) повторяет проверенную логику:
**пейсер запросов** (≈5.9 req/s, разнос стартов 170 мс под лимит ~7 req/s),
**ретраи** с `Retry-After` и экспоненциальным бэкоффом, **кэш ответов** (TTL),
**квантование границы дат** до 10-мин корзин и **пагинацию волнами**.

### Инкрементальная синхронизация (для крупных аккаунтов)

Аккаунт на 16k+ сделок не успевает синхронизироваться за один запрос. Поэтому
sync **порционный и возобновляемый**:

- Курсор хранится в таблице `sync_state` (`organization_id`, `phase`,
  `cursor_pipeline`, `cursor_page`, счётчики, `status`). Фазы:
  `pipelines → leads → events → done`.
- Каждый вызов `POST /api/sync/amocrm` обрабатывает **ограниченную порцию**
  (десятки страниц) с мягким дедлайном ~45с, сохраняет курсор **после каждой
  волны** и возвращает `{ done, phase, progress, leadsSynced, eventsProcessed }`.
  Клиент (`SyncButton`) повторяет вызов, пока не `done`, показывая прогресс-бар.
- Сделки пишутся **идемпотентным upsert** (по `organization_id, source,
  external_id`) — повтор страницы после ретрая не плодит дубли.
- Реконструкция истории (`events`) идёт **отдельной фазой и порциями**: каждая
  волна бьётся в RPC `apply_reached_ranks`, которая поднимает `reached_rank`
  пачкой (без построчных запросов).
- **Таймаут/ошибка amoCRM не роняют процесс 500-й**: курсор уже сохранён,
  маршрут возвращает «resumable»-статус, клиент автоматически повторяет с места
  остановки (до 5 ретраев с бэкоффом).

Окно по умолчанию — `SYNC_WINDOW_DAYS` (**30 дней**); фильтр периода работает
внутри него. Кнопка **«Полная история (365 дн.)»** запускает широкую загрузку
(тоже порциями). «Всё время» в фильтре = всё синхронизированное.

При синхронизации сохраняются **все этапы** воронок (включая системные 142
«успешно» и 143 «слив») как `amocrm_stages` с позиционным `rank`, у каждой
сделки — `status_external_id`, `pipeline_external_id`, `responsible`, а также
справочник менеджеров (`GET /api/v4/users` → `amocrm_users`). По завершении в
лог выводится диагностика: число этапов на воронку, сделок всего и без
`status_id`, число менеджеров и распределение сделок по этапам
(`[sync amocrm] ИТОГО: …` / `распределение по этапам:`).

### Аналитика (`lib/analytics`)

Чистые функции (`computeReport`) считают по `reached_rank`: дошло/конверсия/
потеряно/среднее время на этапе, **узкое место** с текстовым диагнозом, и
ключевые метрики автосалона — **Доходимость** (записан на визит → приехал) и
**Закрытие** (приехал → заказ/предоплата). Этапы для этих метрик
**матчатся по названию динамически** (регистр/пробелы/казахская «і»
нормализуются; названия берутся из воронки конкретной компании, ничего не
захардкожено). Если подходящих этапов нет — метрика помечается недоступной без
падения.

**Производительность (аналитика в SQL).** Все агрегаты считаются **на стороне
Postgres** через RPC (`report_pipelines`, `report_headline`, `report_funnel`,
`report_loss_reasons`, `report_managers`, `report_manager_stages` — миграция
**0008**, `SECURITY DEFINER` + проверка членства, плюс индексы по
`leads(organization_id, …)`). Node получает уже готовые маленькие результаты
(десятки строк), а не все ~16k сделок — дашборд/воронка/менеджеры грузятся за
доли секунды. Логика и цифры те же: позиционный ранг этапа, `reached_rank`,
статусы 142/143, период по `created_at_src`, матчинг ключевых метрик по
названию (на крошечном выходе RPC). Страницы разделены на мгновенную «оболочку»
(шапка + фильтры) и данные за границей `Suspense` — при смене периода/воронки
лоадер показывается только в области цифр, фильтры не блокируются.

**Менеджеры** (`/managers`): по каждому менеджеру — имя (из `amocrm_users`),
число сделок, выиграно/потеряно/в работе, конверсия, сумма выигранных и топ
этапов. Полный список, без обрезки. Краткий топ-5 продублирован на дашборде.

### Состояния без данных

- amoCRM не подключён → «Подключите amoCRM».
- Подключён, но не синхронизирован → «Запустите синхронизацию».
- Синхронизирован, но за период нет сделок → пустое состояние, без ошибок.

Токены читаются и используются **только на сервере**; запросы к amoCRM идут с
сервера Next.js, в браузер секреты не попадают.

## Интеграция Wazzup (переписка)

Подключается **per-organization** (API-ключ в `integrations`, провайдер
`wazzup`). Клиент — `lib/integrations/wazzup.ts` (server-only, троттлинг +
ретраи).

### ⚠️ Важное ограничение Wazzup API v3

**В Wazzup API v3 НЕТ REST-эндпоинта для истории сообщений** — это
подтверждено по официальной документации
(<https://wazzup24.com/help/api-en/>). По REST доступны только справочники:

| Метод | Эндпоинт | Что даёт |
| --- | --- | --- |
| `GET` | `https://api.wazzup24.com/v3/channels` | подключённые каналы |
| `GET` | `https://api.wazzup24.com/v3/users` | пользователи (менеджеры) |

Авторизация — `Authorization: Bearer <api_key>`. **Входящие сообщения
приходят ТОЛЬКО через вебхуки** (`PATCH /v3/webhooks`, подписка
`messagesAndStatuses`). Поэтому историю чатов нельзя «дотянуть» запросом — её
надо принимать вебхуками и хранить у себя. Заглушку вместо данных не делаем.

### Что работает сейчас

- **Проверка соединения** и **синхронизация** каналов и менеджеров:
  `POST /api/sync/wazzup` (OWNER/ROP) → `wazzup_channels` / `wazzup_users`,
  идемпотентный upsert, RLS соблюдается. Диагностика в лог:
  `[sync wazzup] каналов … менеджеров … сообщений 0 (история через вебхуки)`.
- **Аналитика переписок** (`lib/analytics/conversations.ts`) и страница
  **«Переписка»**: число диалогов, новые написавшие лиды, **неотвеченные**
  (клиент написал — менеджер молчит), **среднее время первого ответа**, разрез
  **по менеджерам**, лента диалогов и просмотр истории сообщения-за-сообщением
  (`/conversations/[id]`). Всё считается из таблиц `messages`/`conversations` и
  оживает, как только сообщения начнут поступать.

### Следующий шаг (после деплоя): приём вебхуков

Эндпоинт-приёмник уже готов, но **выключен** (бездействует, пока не задан
секрет):

- `POST /api/webhooks/wazzup/<orgId>` — сохраняет входящие сообщения в
  `conversations`/`messages` (service-role клиент, т.к. вебхуки приходят без
  сессии; жёстко привязан к организации из URL).
- Включение: задать `integrations.config.webhook_secret` (провайдер `wazzup`) и
  указать Wazzup: `PATCH https://api.wazzup24.com/v3/webhooks` с
  `webhooksUri = https://<домен>/api/webhooks/wazzup/<orgId>` и `crmKey =`
  тот же секрет (Wazzup пришлёт его в заголовке `Authorization`, приёмник
  сверит). Realtime/подписку специально пока не включаем.

> Маппинг полей сообщения (inbound/outbound, id) нужно сверить с реальным
> payload вебхука при включении — поэтому приёмник хранит сырой payload (`raw`)
> рядом с каждой строкой.

## Что дальше (вне текущего объёма)

- **Включить вебхуки Wazzup** (приёмник готов) — и переписка с аналитикой
  оживёт в realtime.
- Синхронизация Sipuni → `calls` (коннектор-заглушка описан интерфейсом).
- Ежедневные отчёты в Telegram → `daily_reports`.
- Биллинг поверх заготовленной таблицы `subscriptions`.
- Перенос тяжёлой синхронизации в фоновую очередь/cron для очень крупных
  аккаунтов (сейчас sync выполняется в обработчике маршрута, `maxDuration=300`).

> Безопасность. Секреты интеграций хранятся в `integrations.config` по
> организациям и **никогда не передаются в браузер** (форма показывает лишь
> факт «сохранено»). Сервисный ключ используется только на сервере.
